RAMI4.0 : Architecture View - Workplace-1 Implementation ()
This view shows how the end-to-end process works in workplace-1, including all human, machine and software (communication) steps. One key aspect is that the conveyor belt is triggered via software as a result of the robot arm sending a notification message. There is no physical or human intervention. The advantage of a software implementation is that we can readily enrich the trigger with data about the object itself. Decision: Robot and conveyor should not know each other. therefore workplace-1 should translate events between them. Rationale: RobotArm and Conveyor are modules that should be easily reusable in other contexts, when either of them is not present. Creating hard dependencies between them means they no longer can operate separately from each other, and means changing their implementation in order to support other scenario's. They therefore do not collaborate directly but via Workplace-1 AAS, which encapsulates the specific deployment where robotarm and conveyor are used together in this particular way. Implementation: Communication between modules should happen via OPCUA. Since RobotArm is emiting a notification that an object has been released, i.e. an event, it would be ideal to use OPCUA's pub-sub mechanism. However, it is anticipated that tool support is still limited for this feature of OPCUA, therefore an alternative mechanism is used whereby a OPCUA Client associated with Workplace-1 AAS will monitor the value DropOff Status of RobotArm. When this value changes from '0' to '1', an object dropoff is understood by the Workplace-1 to have taken place. This invokes the function 'Object Drop To Conveyor Start Translation' , which will translate this event into a command 'Conveyor Start' on the Conveyor OPCUA Server.
