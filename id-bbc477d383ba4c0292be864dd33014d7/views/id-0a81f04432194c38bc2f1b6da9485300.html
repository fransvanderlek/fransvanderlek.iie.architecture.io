RAMI4.0 : Architecture View - RobotArm Implementation ()
This view is about describing the robot as I4.0 Components, i.e.: how the physical robot and it's digital counterpart aka Robot AAS become one system. This boiles down to showing what actuators are controlling the robot movements, how these actuators are (ultimately) controlled by a computer which is running the Robot AAS software. Instance- Maintenance & Usage ============== The RobotAAS is visible by being the component that implements Services in the functional layer. These denotes the value that the Robot provides in the digital world and is associated with Robot AAS boundary ( i.e. the I4.0 Component's digital boundary ). This boundary should be modular and standardized so that the I4.0 Robot becomes easily plugged into/outoff the I4.0 ecosystem and we can use standards to communicate thereby lowering the costs of the integrations needed in a Smart Factury since we don't have a strict hierarchy anymore and hence more connections to manage. Therefore not that the RobotAAS services are being supported by the OPCUA Server interface, since we standardize on OPCUA for all communication between I4.0 Components. Other communication is present, in particular MQTT between the Robot Server and arduino. This should be considered an interface internal to the robot, and not to be used outside the robot since is not standardized. The services that the robot provides, digital and physical, are aggregated into a Robot-as-a-service product. Following the RAMI methodology, this is reflecting the fact that these are the serviecs customers that buy the robot are willing to pay for, and therefore the product is part of the business layer. It will specifiy such things as rates, terms of usage, SLA, etc. Instance - Production ============= Currently, development happens through code development which results in packages that can be deployed on the target environment . In other scenario's it is possible to guide the robotarm to it's correct positions and actions, and record that to be replayed by the robot autonomously. In additon, in the production phase, it is possible to show how the robotarm is assembled/constructed from it's parts.
